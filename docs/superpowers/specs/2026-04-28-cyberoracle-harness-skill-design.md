# CyberOracle Development Harness Skill — Design Spec

**Date:** 2026-04-28
**Status:** Approved
**Scope:** A Claude Code skill that orchestrates long-running, incremental development of CyberOracle's 7 milestones using Anthropic's harness patterns.

---

## 1. Overview

### 1.1 Purpose

A single Claude Code skill (`cyberoracle-harness`) that drives autonomous, incremental development of the CyberOracle monorepo across its 7 milestones (M1-M7, ~11 weeks). Invoked per-milestone, it sets up structured artifacts (feature list, progress file, init script) and then runs repeated coding sessions that implement features one at a time with built-in verification.

### 1.2 Key Decisions

| Decision           | Choice                                                                                        | Rationale                                                                                               |
| ------------------ | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Harness type       | Development harness (runtime patterns as reference only)                                      | Primary goal is building CyberOracle; runtime patterns inform implementation but no shared abstractions |
| Testing strategy   | TDD for `packages/core` + `packages/poster`; E2E (Playwright) for `apps/web` + `apps/desktop` | Pure-function territory benefits from test-first; UI benefits from browser automation                   |
| Scope              | Per-milestone invocation (`/cyberoracle-harness M2`)                                          | Existing docs are extremely detailed; per-milestone keeps focus and avoids re-specifying                |
| Agent architecture | 2 agents (initializer + coder)                                                                | Planner is redundant given detailed docs; evaluator adds cost without proportional benefit              |
| Runtime harness    | Design patterns only (no shared abstractions)                                                 | Patterns from Article 3 inform pipeline implementation; premature to scaffold runtime framework         |
| Skill structure    | Single monolithic skill                                                                       | Early-stage project; one file to maintain; refactor later if needed                                     |

### 1.3 Reference Articles

1. **Effective Harnesses for Long-Running Agents** — Initializer/coder pattern, feature lists, incremental progress, progress files, git-based state management
2. **Harness Design for Long-Running Application Development** — Generator/evaluator pattern (adapted: evaluation baked into coder), sprint contracts (adapted: acceptance steps in feature list), context management
3. **Managed Agents: Decoupling the Brain from the Hands** — Runtime patterns: session as append-only log, brain/hand separation, retry with fallback, progress event emission

## 3. Initializer Agent

### 3.2 Feature List Schema

```json
{
  "$schema": "feature-list-schema",
  "milestone": "M2",
  "milestoneTitle": "Web 端 MVP",
  "sourceDocs": ["docs/PRD.md §4.1", "docs/5完整Prompt文件.md"],
  "features": [
    {
      "id": "M2-001",
      "category": "core|poster|web|desktop",
      "description": "One-sentence description of the feature",
      "testStrategy": "unit|playwright|manual",
      "acceptanceSteps": ["Step 1: ...", "Step 2: ...", "Step 3: ..."],
      "priority": 1,
      "dependsOn": [],
      "passes": false
    }
  ]
}
```

**Field semantics:**

- `testStrategy: "unit"` → TDD: write failing test first, implement, verify pass (for `packages/core`, `packages/poster`)
- `testStrategy: "playwright"` → Implement, then verify via Playwright against `acceptanceSteps` (for `apps/web`, `apps/desktop`)
- `testStrategy: "manual"` → Implement, commit with detailed notes for human review
- `dependsOn` → Array of feature IDs that must pass first
- `priority` → Lower number = higher priority; coding agent picks highest-priority failing feature

### 3.3 Constraint

The initializer **never writes code**. It only generates artifacts: `feature_list.json`, `init.sh`, `progress.md`.

## 4. Coding Agent

The coding agent runs **in repeated sessions**, one feature per session.

### 4.1 Session Startup Sequence

Every session begins with these steps, in order:

1. `pwd` — confirm working directory
2. Read `progress.md` — understand prior session work
3. Read `feature_list.json` — find highest-priority feature where `passes: false`
4. Run `init.sh` — start dev servers for current milestone
5. Run smoke test — verify basic app functionality (catches environment rot)
6. Only then start implementing the next feature

### 4.2 Feature Implementation Loop

```
For each session:
  1. Pick next failing feature (by priority, respecting dependsOn)
  2. IF testStrategy == "unit":
       a. Write failing test
       b. Implement minimum to pass
       c. Refactor
       d. Run test suite, confirm pass
  3. IF testStrategy == "playwright":
       a. Implement feature
       b. Start dev server
       c. Run Playwright verification against acceptanceSteps
       d. Fix any failures
  4. IF testStrategy == "manual":
       a. Implement feature
       b. Commit with detailed notes
  5. Set feature.passes = true ONLY if verification passes
  6. git commit with message: "[M{X}-{id}] {description}"
  7. Update progress.md with session summary
  8. STOP after 1 feature
```

### 4.3 Discipline Rules

- **One feature per session.** Prevents the "try to do too much at once" failure mode (Article 1). If a feature is too large to complete in one session, the agent commits partial progress, marks the feature as `passes: false`, and notes the remaining work in `progress.md` for the next session to continue.
- **Never remove or edit features.** Only change `passes` from `false` to `true`. Prevents declaring victory by deleting work.
- **Leave environment clean.** Every session ends with clean git state. If stuck, revert to last good commit.
- **JSON for feature list.** Model is less likely to inappropriately edit JSON vs Markdown (Article 1 finding).

### 4.4 Session End Artifacts

| Artifact            | Action                                                       |
| ------------------- | ------------------------------------------------------------ |
| `progress.md`       | Append: session date, feature id, what was done, what failed |
| `feature_list.json` | Flip one `passes: false` → `true`                            |
| Git log             | One commit: `[M{X}-{id}] {description}`                      |

### 4.5 Context Management

- **Within-session:** Use Claude Code's built-in compaction for context growth
- **Cross-session:** `progress.md` + `feature_list.json` + git log provide full recovery context
- **No explicit context resets** — Opus 4.6+ doesn't exhibit the "context anxiety" behavior that required resets in Sonnet 4.5 (Article 2 finding)

## 5. Evaluation Criteria

Built into feature acceptance steps, not a separate evaluator agent.

| Criterion            | Applies to        | Threshold                                        | How Verified                   |
| -------------------- | ----------------- | ------------------------------------------------ | ------------------------------ |
| Schema compliance    | `packages/core`   | Zod validation passes on all LLM output samples  | Unit test                      |
| Content safety       | `packages/core`   | No output triggers keyword blacklist             | Unit test                      |
| Render correctness   | `packages/poster` | Snapshot matches golden image (pixel diff < 1%)  | Unit test (pixelmatch + pngjs) |
| SSE streaming        | `apps/server`     | Chunks arrive in order, connection survives 30s+ | Playwright                     |
| Performance budget   | `apps/web`        | LCP ≤ 2.5s, JS gzip ≤ 220KB                      | Playwright + Lighthouse        |
| Type safety          | All packages      | `pnpm typecheck` zero errors                     | CLI verification               |
| Cross-platform build | `apps/desktop`    | `cargo build` succeeds on all 4 targets          | CI verification                |

## 6. Runtime Patterns Reference

Patterns from Article 3 that the coding agent applies when implementing pipeline features. Stored in `references/runtime-patterns.md`.

### 6.1 Session as Append-Only Log

- SSE endpoints (`/api/analyze`, `/api/quick-ask`) log each step to `/results/{id}.status`
- On failure, status file enables recovery logic to resume from exact point
- Applied in: `apps/server` route handlers, `apps/desktop/src-tauri/src/commands/analyze_stream.rs`

### 6.2 Brain/Hand Separation for LLM Calls

- Orchestration logic decoupled from individual LLM API calls
- `PipelineRunner` pattern: each step is a named operation with input/output; runner handles retry, timeout, error propagation
- Applied in: two-phase prompt pipeline (VLM → LLM), daily fortune pipeline

### 6.3 Progress Event Emission

- Rust side: `tauri::Window::emit("analyze:chunk", payload)` (per PRD §5.5)
- Server side: SSE chunks include step identifiers
- Event schema: `{ step: "vlm_observe" | "llm_interpret" | "poster_render", status: "running" | "done" | "error", data: ... }`

### 6.4 Retry with Degraded Fallback

- Primary (通义千问 VL-Max) → fallback (GLM-4V) → ultimate fallback (GPT-4o) — per PRD §4.4
- Each retry logged to status file
- Applied in: `apps/server` LLM proxy layer

## 7. Phased Build Plan

### Phase 1: Core Skill (Current Session)

- Write `skill.md` with initializer + coding agent prompts
- Write `templates/feature-list.json` with schema and sample M2 feature list
- Write `templates/init-sh.sh` and `templates/progress.md`
- Write `references/runtime-patterns.md`
- Validate skill loads and initializer produces correct artifacts

### Phase 2: Milestone Configs (Per Milestone, As Needed)

- Invoke `/cyberoracle-harness M1` → initializer generates M1 feature list
- Each invocation is a live test of the skill
- Adjust prompts based on observed behavior

### Phase 3: Stabilization (Post-M3)

- Extract repeated patterns from generated feature lists into skill
- Consider refactoring into generated `tools/harness/` scripts if monolithic skill gets unwieldy
- Add dedicated runtime harness skill if needed

## 8. Failure Modes and Mitigations

Adapted from Article 1's failure mode table:

| Problem                                       | Mitigation in This Skill                                                                                      |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Agent declares victory too early              | Feature list with explicit `passes` field; strongly-worded "never remove features" constraint                 |
| Agent leaves environment broken               | Session ends with clean git state; next session starts with smoke test; revert if needed                      |
| Agent marks features as done prematurely      | Verification required: unit tests for core/poster, Playwright for apps; only flip `passes` after verification |
| Agent wastes time figuring out how to run app | `init.sh` generated per-milestone; coding agent reads it at session start                                     |
| Context fills during long feature             | One feature per session; compaction handles within-session growth; `progress.md` bridges sessions             |
| Generated code has subtle bugs                | Playwright verification tests as a human would (Article 1 finding); snapshot tests for poster output          |
